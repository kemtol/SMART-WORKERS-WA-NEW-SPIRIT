# New Spirit Scrapper

WhatsApp Ops worker for reading movement messages from the `New Spirit` group, parsing MVT departure/arrival formats, storing them in SQLite, and syncing final movement rows to Google Sheets.

## Current Data Flow

```text
WhatsApp group
  -> Baileys listener
  -> Python ingest service
  -> SQLite raw_messages / flight_movements
  -> Google Sheets tab: Movements_Internal
```

`Movements_Internal` is the current implemented Sheets tab. The PRD below defines the next target data model with `RAW`, `FLIGHT_RAW`, and `FLIGHT_OPS`. Older experimental tabs such as `Schedules` or `Movements` should not be used.

## Product Requirements: AI-Cleaned Flight Ops Dataset

This section is the PRD for the next data-product version. The current implementation already covers WhatsApp ingestion, SQLite storage, rule-based parsing, and Sheets sync. The target product adds a bronze/silver/gold dataset model and an AI deep-cleaning gate.

### Objective

Turn noisy WhatsApp operational messages into a clean flight operations dataset that can be trusted for daily ops review, reporting, and downstream automation.

The system must preserve raw WhatsApp messages, keep deterministic parser output for auditability, and publish an AI-cleaned operational layer only after validation gates pass.

### Dataset Layers

Google Sheets should expose three main tabs:

```text
RAW         bronze dataset: raw WhatsApp messages
FLIGHT_RAW  silver dataset: rule-based parsed flight movements
FLIGHT_OPS  gold dataset: AI-cleaned and validated operational rows
```

`RAW` is append-only and represents the source of truth. It should not be edited manually.

`FLIGHT_RAW` is generated by the rule-based parser. It may contain incomplete or imperfect values, but every value should be traceable back to a raw message and parser version.

`FLIGHT_OPS` is the final operational dataset. A row can enter this tab only after AI deep-cleaning and validation have completed successfully.

### Pipeline

```text
1. Connect, read, and store
   WhatsApp group -> raw_messages table -> RAW tab

2. Parse and normalize
   raw_messages -> flight_movements table -> FLIGHT_RAW tab

3. AI deep-cleaning
   FLIGHT_RAW pending rows -> prompt endpoint -> validation gates -> FLIGHT_OPS tab
```

The raw message must always remain available so the parser and AI cleaner can be rerun or audited later.

### Deep-Cleaning Schedule

The AI deep-cleaning worker should run every 15 minutes by default.

It should also support manual trigger modes:

- Run deep-cleaning now for all eligible rows.
- Run deep-cleaning for one selected row.
- Force re-check a row that was already cleaned.

The worker must not process a row again once it has been successfully cleaned, unless the row is explicitly marked for force check.

For the initial development phase, AI deep-cleaning can be Codex-assisted and manually triggered. Once the prompt, schema, and validation gates are stable, the same contract should move into a dedicated worker that calls the selected prompt endpoint.

### Row Status Model

`FLIGHT_RAW` should include deep-cleaning control fields:

```text
deepclean_status
deepclean_force_check
deepclean_requested_at
deepcleaned_at
deepclean_prompt_version
deepclean_model
deepclean_error
flight_ops_id
```

Recommended `deepclean_status` values:

```text
pending
cleaning
cleaned
failed
needs_review
skipped
```

Processing rules:

- `pending` rows are eligible for the next scheduled deep-cleaning run.
- `cleaning` means the worker is currently processing the row.
- `cleaned` rows are not processed again by normal scheduled runs.
- `failed` rows may be retried by a later run, with retry limits.
- `needs_review` rows require human review before becoming operational gold data.
- `deepclean_force_check = TRUE` makes the row eligible even if it is already `cleaned`.
- After a successful forced re-check, the worker must set `deepclean_force_check = FALSE`.

### Force Check UX

For Google Sheets, the simplest force-check mechanism is a checkbox column on `FLIGHT_RAW`:

```text
deepclean_force_check
```

Ops users can tick the checkbox on a row. The next scheduled run, or a manual trigger, should reprocess that row and update `FLIGHT_OPS`.

The row should retain audit metadata showing the prompt version, model, and latest cleaned timestamp.

### AI Deep-Cleaning Input

The prompt endpoint should receive enough context to clean the row without losing auditability:

```text
raw_message_id
movement_id
source_text
rule_based_fields
airport_master_context
previous_flight_ops_row, if force-checking an existing gold row
prompt_version
```

The AI should treat the source text as authoritative and use rule-based fields as hints, not as absolute truth.

### AI Deep-Cleaning Output

The prompt endpoint must return strict JSON. No prose should be accepted as a successful response.

The gold output should include normalized fields such as:

```text
operation_date
movement_type
registration
aircraft_type
flight_seq
leg_origin_code
leg_destination_code
route_full
takeoff_time
eta_time
ata_time
pax
pax_weight_kg
baggage_kg
cargo_kg
total_load_kg
remark
ops_status
ai_confidence
review_notes
```

Exact schema can evolve, but each schema version must be explicit and traceable.

### Validation Gates

A row can be written to `FLIGHT_OPS` only if these gates pass:

- AI response is valid JSON.
- Required fields match the expected schema version.
- Registration format is valid, for example `PK-SNW`.
- Movement type is recognized, for example `departure` or `arrival`.
- Airport codes can be matched to the internal airport master or accepted as explicitly unmapped.
- Time fields are normalized consistently.
- Numeric load fields are parsed as numbers where possible.
- AI confidence is above the configured threshold.
- The original `raw_message_id` and `movement_id` are retained.

If gates fail, the row should be marked `failed` or `needs_review`, not silently promoted to `FLIGHT_OPS`.

### Google Sheets Behavior

The target Sheets workbook should contain:

- `RAW`: raw WhatsApp messages.
- `FLIGHT_RAW`: parser output plus deep-cleaning status columns.
- `FLIGHT_OPS`: AI-cleaned gold data.

Scheduled workers should append/update these tabs without requiring manual copy-paste.

Manual edits should be limited to operational control fields such as:

- `deepclean_force_check`
- optional human review fields
- optional ops notes

Raw text and generated parser fields should remain system-owned.

### Reprocessing Rules

Default behavior:

```text
deepclean_status = cleaned
deepclean_force_check = FALSE
```

means the row is skipped.

Allowed reprocessing cases:

- `deepclean_force_check = TRUE`
- prompt version migration run
- schema migration run
- manual admin override

Reprocessing must not destroy audit history. At minimum, store latest prompt version/model/timestamp. Preferably, keep a historical table for previous AI outputs.

### Non-Goals

The first version does not need to:

- replace the rule-based parser completely
- edit WhatsApp messages
- infer data that is not supported by the source text or airport master
- auto-delete raw messages
- expose public endpoints without token protection

### Acceptance Criteria

The product is ready for initial ops usage when:

- New WhatsApp messages appear in `RAW`.
- Rule-based parsed rows appear in `FLIGHT_RAW`.
- Every eligible `FLIGHT_RAW` row is deep-cleaned within 15 minutes.
- Successfully cleaned rows appear in `FLIGHT_OPS`.
- Already cleaned rows are skipped by default.
- Checking `deepclean_force_check` reprocesses that row.
- Failed AI responses do not pollute `FLIGHT_OPS`.
- Every `FLIGHT_OPS` row links back to `movement_id`, `raw_message_id`, and `source_text`.
- Prompt version and AI model are recorded for every gold row.

## WhatsApp Listener

Use a dedicated WhatsApp bot number for this worker. Do not share the same linked-device session with another WhatsApp Web client.

Run once:

```bash
npm run listen
```

On first run, scan the QR with:

```text
WhatsApp -> Linked devices -> Link a device
```

For long-running operation, use the restart loop:

```bash
npm run listen:loop
```

```text

```

## Movement Parser

The parser is in `app/movement_parser.py`. It understands the current Ops message patterns:

- `MVT Dept`
- `Mvt Arrival`
- aircraft registration, e.g. `PK-SNW`
- aircraft type, e.g. `C208B-EX`
- daily flight sequence, e.g. `Flight 04`
- multi-leg routes, e.g. `AAP-RTU-AAP`
- `Engine Start`, `Take Off`, `ETA`, `ATA`
- pax, pax weight, baggage, cargo, total load

Departure routes are split into one row per leg. For example:

```text
AAP-RTU-AAP
```

becomes:

```text
AAP -> RTU
RTU -> AAP
```

Rebuild parsed movement rows from existing raw messages:

```bash
npm run movements:rebuild
```

## Google Sheets Sync

The current implemented Sheets destination is:

```text
Movements_Internal
```

The PRD target replaces this with `RAW`, `FLIGHT_RAW`, and `FLIGHT_OPS`.

Set up Apps Script:

1. Open the Google Sheet.
2. Go to `Extensions` -> `Apps Script`.
3. Paste `integrations/google-sheets-webhook.gs`.
4. Add Script Properties:
   - `TOKEN`: private random token
   - `SPREADSHEET_ID`: target spreadsheet ID
5. Deploy as a Web App.
6. Set `Execute as` to `Me`.
7. Set access to `Anyone`.
8. Copy the `/exec` Web App URL.

Create local config:

```bash
cp config/google-sheets.env.example config/google-sheets.env
```

Set:

```bash
GOOGLE_SHEETS_WEBHOOK_URL=https://script.google.com/macros/s/YOUR_DEPLOYMENT_ID/exec
GOOGLE_SHEETS_WEBHOOK_TOKEN=the-same-token-from-apps-script
GOOGLE_SHEETS_TAB=Movements_Internal
```

Backfill movement rows once:

```bash
npm run sheets:sync
```

Run continuous sync:

```bash
npm run sheets:sync:loop
```

Sheets sync state is stored in:

```text
data/google-sheets-movement-sync-state.json
```

Delete this state file only if you intentionally want to append all movement rows again.

## Runtime Data Migration

For moving to another server, do not commit raw WhatsApp runtime data to a public GitHub repo. The SQLite database can contain raw messages, sender IDs, timestamps, and operational details.

Use a local migration archive instead.

Export runtime data:

```bash
bin/export-migration.sh
```

This includes, when present:

- `data/ops_messages.sqlite3`
- `data/reference/master_iata.json`
- `data/google-sheets-movement-sync-state.json`

Encrypt the archive before moving it:

```bash
MIGRATION_PASSPHRASE='use-a-long-passphrase' bin/export-migration.sh
```

Import on the new server:

```bash
MIGRATION_PASSPHRASE='use-a-long-passphrase' bin/import-migration.sh /path/to/new-spirit-runtime-YYYYMMDDTHHMMSSZ.tar.gz.enc
```

Local Google Sheets env can also be included if needed:

```bash
INCLUDE_LOCAL_ENV=1 MIGRATION_PASSPHRASE='use-a-long-passphrase' bin/export-migration.sh
```

WhatsApp linked-device credentials under `.runtime-auth/` are not included by this migration script. Re-link the worker with a fresh QR on the new machine.

## Optional History Sync

`src/scrape-history.js` is experimental. WhatsApp decides how much old history a linked device receives, so manual export remains the more reliable archive path.

If using copied credentials from another WhatsApp Web integration, set:

```bash
OPENCLAW_WHATSAPP_AUTH_DIR=/path/to/source/auth
```

The project copies credentials into `.runtime-auth/` and does not need to modify the source credential directory.
